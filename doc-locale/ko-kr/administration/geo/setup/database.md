이 문서에서는 프라이머리 GitLab 데이터베이스를 세컨더리 사이트의 데이터베이스로 복제하기 위한 최소 필수 단계를 설명합니다. 데이터베이스의 설정 및 크기를 포함한 특성에 따라 일부 값을 변경해야 할 수 있습니다.

> [!note]
> GitLab 설치가 외부 PostgreSQL 인스턴스(Linux 패키지 설치로 관리되지 않음)를 사용하는 경우 역할이 모든 필요한 구성 단계를 수행할 수 없습니다. 이 경우 대신 [외부 PostgreSQL 인스턴스를 사용한 Geo](external_database.md) 프로세스를 사용하세요.

**세컨더리** 사이트가 **프라이머리** 사이트와 동일한 버전의 GitLab Enterprise Edition을 실행 중인지 확인하세요. **프라이머리** 사이트에 [Premium 또는 Ultimate 구독](https://about.gitlab.com/pricing/) 라이선스를 추가했는지 확인하세요.

테스트 또는 프로덕션 환경에서 실행하기 전에 이 모든 단계를 읽고 검토하세요.

> [!note]
> 설정 프로세스의 단계는 문서화된 순서대로 완료해야 합니다. 그렇지 않으면 진행하기 전에 [모든 이전 단계를 완료](_index.md#using-linux-package-installations)하세요.

## 데이터베이스 암호 일관성 요구 사항 {#database-password-consistency-requirements}

각 데이터베이스 관련 암호 유형은 모든 Geo 사이트(프라이머리 및 세컨더리)에서 일치하는 값을 가져야 합니다. 여기에는 다음이 포함됩니다:

- `postgresql['sql_replication_password']` (복제 사용자 암호, MD5)
- `postgresql['sql_user_password']` (GitLab 데이터베이스 사용자 암호, MD5)
- `gitlab_rails['db_password']` (GitLab 데이터베이스 사용자 암호, 평문)
- `patroni['replication_password']` (Patroni 설정의 경우, 평문)
- `patroni['password']` (Patroni API 인증의 경우, 평문)
- `postgresql['pgbouncer_user_password']` (PgBouncer 사용 시, MD5)

예를 들어 프라이머리 사이트에서 구성된 `patroni['password']` 값은 모든 세컨더리 사이트의 `patroni['password']` 값과 동일해야 합니다.

이러한 암호는 프라이머리 및 세컨더리 사이트 간의 데이터베이스 인증 및 복제에 사용됩니다. 다른 암호를 사용하면 복제 실패가 발생하고 Geo가 제대로 작동하지 않습니다.

## 단일 인스턴스 데이터베이스 복제 {#single-instance-database-replication}

단일 인스턴스 데이터베이스 복제는 설정하기가 더 쉽고 클러스터된 대안과 동일한 Geo 기능을 제공합니다. 단일 머신에서 실행되거나 향후 클러스터된 설치를 위해 Geo를 평가하려는 설정에 유용합니다.

단일 인스턴스는 Patroni를 사용하여 클러스터된 버전으로 확장할 수 있으며, 이는 고가용성 아키텍처에 권장됩니다.

아래 지침에 따라 PostgreSQL 복제를 단일 인스턴스 데이터베이스로 설정하는 방법을 알아봅니다. 또는 Patroni 클러스터를 사용한 복제 설정에 대한 [다중 노드 데이터베이스 복제](#multi-node-database-replication) 지침을 참조할 수 있습니다.

### PostgreSQL 복제 {#postgresql-replication}

쓰기 작업이 발생하는 GitLab **프라이머리** 사이트는 **프라이머리** 데이터베이스 서버에 연결됩니다. **세컨더리** 사이트는 자신의 데이터베이스 서버(읽기 전용)에 연결됩니다.

[PostgreSQL의 복제 슬롯](https://medium.com/@tk512/replication-slots-in-postgresql-b4b03d277c75)을 사용하여 **프라이머리** 사이트가 **세컨더리** 사이트가 복구하는 데 필요한 모든 데이터를 유지하도록 해야 합니다. 자세한 내용은 아래를 참조하세요.

다음 가이드에서는 다음을 가정합니다:

- Linux 패키지를 사용하고 있으므로 PostgreSQL 12 이상을 사용하고 있으며, 여기에는 [`pg_basebackup` 도구](https://www.postgresql.org/docs/16/app-pgbasebackup.html)가 포함됩니다.
- **프라이머리** 사이트가 이미 설정되어 있고(복제하는 GitLab 서버), Linux 패키지 설치로 관리되는 PostgreSQL(또는 해당 버전)을 실행 중이며, 동일한 [PostgreSQL 버전](../_index.md#requirements-for-running-geo), OS 및 GitLab이 설정된 새 **세컨더리** 사이트가 있습니다.

> [!warning]
> Geo는 스트리밍 복제와 함께 작동합니다. 논리적 복제는 지원되지 않지만 [에픽 18022](https://gitlab.com/groups/gitlab-org/-/epics/18022)는 이 동작을 변경할 것을 제안합니다.

#### 1단계. **프라이머리** 사이트 구성 {#step-1-configure-the-primary-site}

1. GitLab **프라이머리** 사이트로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. GitLab을 업그레이드할 때 예기치 않은 다운타임을 피하기 위해 [자동 PostgreSQL 업그레이드를 거부합니다](https://docs.gitlab.com/omnibus/settings/database/#opt-out-of-automatic-postgresql-upgrades). Geo를 사용한 PostgreSQL 업그레이드에 관한 알려진 [주의사항](https://docs.gitlab.com/omnibus/settings/database/#caveats-when-upgrading-postgresql-with-geo)을 인식하세요. 특히 더 큰 환경의 경우 PostgreSQL 업그레이드를 신중하게 계획하고 실행해야 합니다. 그 결과 향후로 PostgreSQL 업그레이드가 정기적인 유지보수 활동의 일부가 되도록 합니다.
1. `/etc/gitlab/gitlab.rb`을 편집하고 사이트의 **unique** 이름을 추가하세요:

   ```ruby
   ##
   ## The unique identifier for the Geo site. See
   ## https://docs.gitlab.com/administration/geo_sites/#common-settings
   ##
   gitlab_rails['geo_node_name'] = '<site_name_here>'
   ```

1. **프라이머리** 사이트를 다시 구성하여 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl reconfigure
   ```

1. 아래 명령을 실행하여 사이트를 **프라이머리** 사이트로 정의하세요:

   ```shell
   gitlab-ctl set-geo-primary-node
   ```

   이 명령은 `/etc/gitlab/gitlab.rb`에서 정의한 `external_url`을 사용합니다.

1. `gitlab` 데이터베이스 사용자의 암호를 정의하세요:

   원하는 비밀번호의 MD5 해시를 생성합니다:

   ```shell
   gitlab-ctl pg-password-md5 gitlab
   # Enter password: <your_db_password_here>
   # Confirm password: <your_db_password_here>
   # fca0b89a972d69f00eb3ec98a5838484
   ```

   `/etc/gitlab/gitlab.rb`을(를) 편집합니다:

   ```ruby
   # Fill with the hash generated by `gitlab-ctl pg-password-md5 gitlab`
   postgresql['sql_user_password'] = '<md5_hash_of_your_db_password>'

   # Every node that runs Puma or Sidekiq needs to have the database
   # password specified as below. If you have a high-availability setup, this
   # must be present in all application nodes.
   gitlab_rails['db_password'] = '<your_db_password_here>'
   ```

1. 데이터베이스 [복제 사용자](https://www.postgresql.org/docs/16/warm-standby.html#STREAMING-REPLICATION)의 비밀번호를 정의합니다.

   `/etc/gitlab/gitlab.rb`의 `postgresql['sql_replication_user']` 설정 아래에 정의된 사용자 이름을 사용합니다. 기본값은 `gitlab_replicator`입니다. 사용자 이름을 다른 것으로 변경한 경우 아래 지침을 조정하세요.

   원하는 비밀번호의 MD5 해시를 생성합니다:

   ```shell
   gitlab-ctl pg-password-md5 gitlab_replicator
   # Enter password: <your_replication_password_here>
   # Confirm password: <your_replication_password_here>
   # 950233c0dfc2f39c64cf30457c3b7f1e
   ```

   `/etc/gitlab/gitlab.rb`을(를) 편집합니다:

   ```ruby
   # Fill with the hash generated by `gitlab-ctl pg-password-md5 gitlab_replicator`
   postgresql['sql_replication_password'] = '<md5_hash_of_your_replication_password>'
   ```

   Linux 패키지 설치로 관리되지 않는 외부 데이터베이스를 사용하는 경우 `gitlab_replicator` 사용자를 만들고 수동으로 해당 사용자의 암호를 정의해야 합니다:

   ```sql
   --- Create a new user 'replicator'
   CREATE USER gitlab_replicator;

   --- Set/change a password and grants replication privilege
   ALTER USER gitlab_replicator WITH REPLICATION ENCRYPTED PASSWORD '<replication_password>';
   ```

1. `/etc/gitlab/gitlab.rb`을 편집하고 역할을 `geo_primary_role`로 설정하세요(자세한 내용은 [Geo 역할](https://docs.gitlab.com/omnibus/roles/#gitlab-geo-roles) 참조):

   ```ruby
   ## Geo Primary role
   roles(['geo_primary_role'])
   ```

1. PostgreSQL을 네트워크 인터페이스에서 수신 대기하도록 구성합니다:

   보안상의 이유로 PostgreSQL은 기본적으로 네트워크 인터페이스에서 수신 대기하지 않습니다. 그러나 Geo에서는 **세컨더리** 사이트가 **프라이머리** 사이트의 데이터베이스에 연결할 수 있어야 합니다. 이러한 이유로 각 사이트의 IP 주소가 필요합니다.

   > [!note]
   > 외부 PostgreSQL 인스턴스의 경우 [추가 지침](external_database.md)을 참조하세요.

   클라우드 제공자를 사용하는 경우 클라우드 제공자의 관리 콘솔을 통해 각 Geo 사이트의 주소를 조회할 수 있습니다.

   Geo 사이트의 주소를 조회하려면 Geo 사이트에 SSH로 연결하고 실행합니다:

   ```shell
   ##
   ## Private address
   ##
   ip route get 255.255.255.255 | awk '{print "Private address:", $NF; exit}'

   ##
   ## Public address
   ##
   echo "External address: $(curl --silent "ipinfo.io/ip")"
   ```

   대부분의 경우 다음 주소를 사용하여 GitLab Geo를 구성합니다:

   | 구성                           | 주소                                                               |
   |:----------------------------------------|:----------------------------------------------------------------------|
   | `postgresql['listen_address']`          | **프라이머리** 사이트의 공용 또는 VPC 프라이빗 주소입니다.                     |
   | `postgresql['md5_auth_cidr_addresses']` | **프라이머리** 및 **세컨더리** 사이트의 공용 또는 VPC 프라이빗 주소입니다. |

   Google Cloud Platform, SoftLayer 또는 가상 프라이빗 클라우드(VPC)를 제공하는 다른 공급자를 사용하는 경우 `postgresql['md5_auth_cidr_addresses']` 및 `postgresql['listen_address']`에 대해 **프라이머리** 및 **세컨더리** 사이트의 "프라이빗" 또는 "내부" 주소를 사용하는 것을 권장합니다.

   `listen_address` 옵션은 주어진 주소에 해당하는 인터페이스를 통해 네트워크 연결을 위해 PostgreSQL을 엽니다. 자세한 내용은 [PostgreSQL 문서](https://www.postgresql.org/docs/16/runtime-config-connection.html)를 참조하세요.

   > [!note]
   > `0.0.0.0` 또는 `*`를 `listen_address`로 사용해야 하는 경우 `127.0.0.1`를 통해 Rails가 연결할 수 있도록 `postgresql['md5_auth_cidr_addresses']` 설정에 `127.0.0.1/32`를 추가해야 합니다. 자세한 내용은 [문제 5258](https://gitlab.com/gitlab-org/omnibus-gitlab/-/issues/5258)을 참조하세요.

   네트워크 구성에 따라 제안된 주소가 잘못될 수 있습니다. **프라이머리** 및 **세컨더리** 사이트가 로컬 영역 네트워크를 통해 연결하거나 [Amazon VPC](https://aws.amazon.com/vpc/) 또는 [Google VPC](https://cloud.google.com/vpc/)와 같은 가용성 영역을 연결하는 가상 네트워크를 통해 연결하는 경우 `postgresql['md5_auth_cidr_addresses']`에 대해 **세컨더리** 사이트의 프라이빗 주소를 사용해야 합니다.

   `/etc/gitlab/gitlab.rb`을 편집하고 IP 주소를 네트워크 구성에 적절한 주소로 바꾸면서 다음을 추가하세요:

   ```ruby
   ##
   ## Primary address
   ## - replace '<primary_node_ip>' with the public or VPC address of your Geo primary node
   ##
   postgresql['listen_address'] = '<primary_site_ip>'

   ##
   # Allow PostgreSQL client authentication from the primary and secondary IPs. These IPs may be
   # public or VPC addresses in CIDR format, for example ['198.51.100.1/32', '198.51.100.2/32']
   ##
   postgresql['md5_auth_cidr_addresses'] = ['<primary_site_ip>/32', '<secondary_site_ip>/32']

   ##
   ## Replication settings
   ##
   # postgresql['max_replication_slots'] = 1 # Set this to be the number of Geo secondary nodes if you have more than one
   # postgresql['max_wal_senders'] = 10
   # postgresql['wal_keep_segments'] = 10
   ```

1. PostgreSQL이 다시 시작되고 개인 주소에서 수신 대기할 때까지 자동 데이터베이스 마이그레이션을 일시적으로 비활성화합니다. `/etc/gitlab/gitlab.rb`을 편집하고 구성을 false로 변경하세요:

   ```ruby
   ## Disable automatic database migrations
   gitlab_rails['auto_migrate'] = false
   ```

1. 선택사항. 다른 **세컨더리** 사이트를 추가하려면 관련 설정이 다음과 같이 보입니다:

   ```ruby
   postgresql['md5_auth_cidr_addresses'] = ['<primary_site_ip>/32', '<secondary_site_ip>/32', '<another_secondary_site_ip>/32']
   ```

   또한 `wal_keep_segments` 및 `max_wal_senders`를 편집하여 데이터베이스 복제 요구 사항을 충족하도록 할 수 있습니다. 자세한 내용은 [PostgreSQL - 복제 문서](https://www.postgresql.org/docs/16/runtime-config-replication.html)를 참조하세요.

1. 파일을 저장하고 GitLab을 다시 구성하여 데이터베이스 수신 변경 사항과 복제 슬롯 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl reconfigure
   ```

   PostgreSQL을 다시 시작하여 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl restart postgresql
   ```

1. PostgreSQL이 다시 시작되고 프라이빗 주소에서 수신 대기하므로 마이그레이션을 다시 활성화하세요.

   `/etc/gitlab/gitlab.rb`을 편집하고 구성을 `true`로 **변경**하세요:

   ```ruby
   gitlab_rails['auto_migrate'] = true
   ```

   파일을 저장하고 GitLab을 재구성합니다:

   ```shell
   gitlab-ctl reconfigure
   ```

1. 이제 PostgreSQL 서버가 원격 연결을 수락하도록 설정되었으므로 `netstat -plnt | grep 5432`을 실행하여 PostgreSQL이 **프라이머리** 사이트의 프라이빗 주소에서 포트 `5432`을 수신 대기하는지 확인하세요.
1. GitLab이 다시 구성될 때 인증서가 자동으로 생성되었습니다. 이는 PostgreSQL 트래픽을 도청자로부터 보호하는 데 자동으로 사용됩니다. 능동형("중간자") 공격자로부터 보호하려면 **세컨더리** 사이트에 인증서에 서명한 CA의 사본이 필요합니다. 이 자체 서명 인증서의 경우 다음 명령을 실행하여 **프라이머리** 사이트의 PostgreSQL `server.crt` 파일의 사본을 만드세요:

   ```shell
   cat ~gitlab-psql/data/server.crt
   ```

   출력을 클립보드 또는 로컬 파일로 복사하세요. **세컨더리** 사이트를 설정할 때 필요합니다! 인증서는 민감한 데이터가 아닙니다.

   그러나 이 인증서는 일반적인 `PostgreSQL` 일반 이름으로 만들어집니다. 이를 위해 데이터베이스를 복제할 때 `verify-ca` 모드를 사용해야 하며, 그렇지 않으면 호스트 이름 불일치로 인해 오류가 발생합니다.

1. 선택사항. 생성된 인증서 대신 자신의 SSL 인증서를 생성하고 수동으로 [PostgreSQL용 SSL 구성](https://docs.gitlab.com/omnibus/settings/database/#configuring-ssl)하세요.

   최소한 SSL 인증서와 키가 필요합니다. `postgresql['ssl_cert_file']` 및 `postgresql['ssl_key_file']` 값을 데이터베이스 SSL 문서에 따라 전체 경로로 설정하세요.

   이를 통해 데이터베이스를 복제할 때 `verify-full` SSL 모드를 사용할 수 있고 CN에서 전체 호스트 이름을 확인하는 추가 이점을 얻을 수 있습니다.

   앞으로는 이전에 자동으로 생성된 자체 서명 인증서 대신 `postgresql['ssl_cert_file']`에도 설정한 이 인증서를 사용할 수 있습니다. 이를 통해 CN이 일치하는 경우 `verify-full`를 복제 오류 없이 사용할 수 있습니다.

   프라이머리 데이터베이스의 `/etc/gitlab/gitlab.rb`을 열고 `postgresql['ssl_ca_file']`(CA 인증서)를 검색하세요. 값을 클립보드에 복사하여 나중에 `server.crt`에 붙여넣으세요.

#### 2단계. **세컨더리** 서버 구성 {#step-2-configure-the-secondary-server}

1. GitLab **세컨더리** 사이트로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. GitLab을 업그레이드할 때 예기치 않은 다운타임을 피하기 위해 [자동 PostgreSQL 업그레이드를 거부합니다](https://docs.gitlab.com/omnibus/settings/database/#opt-out-of-automatic-postgresql-upgrades). Geo를 사용한 PostgreSQL 업그레이드에 관한 알려진 [주의사항](https://docs.gitlab.com/omnibus/settings/database/#caveats-when-upgrading-postgresql-with-geo)을 인식하세요. 특히 더 큰 환경의 경우 PostgreSQL 업그레이드를 신중하게 계획하고 실행해야 합니다. 그 결과 향후로 PostgreSQL 업그레이드가 정기적인 유지보수 활동의 일부가 되도록 합니다.
1. 애플리케이션 서버 및 Sidekiq를 중지하세요:

   ```shell
   gitlab-ctl stop puma
   gitlab-ctl stop sidekiq
   ```

   > [!note]
   > 이 단계는 사이트가 완전히 구성되기 전에 아무것도 실행하려고 하지 않도록 하는 데 중요합니다.

1. [TCP 연결 확인](../../raketasks/maintenance.md)을 **프라이머리** 사이트의 PostgreSQL 서버에 수행하세요:

   ```shell
   gitlab-rake gitlab:tcp_check[<primary_site_ip>,5432]
   ```

   > [!note]
   > 이 단계가 실패하면 잘못된 IP 주소를 사용하고 있거나 방화벽이 사이트에 대한 액세스를 방지하고 있을 수 있습니다. 공용 주소와 개인 주소의 차이점에 주의하여 IP 주소를 확인합니다. 방화벽이 있는 경우 **세컨더리** 사이트가 **프라이머리** 사이트에 포트 5432로 연결할 수 있도록 허용되는지 확인하세요.

1. **세컨더리** 사이트에 `server.crt` 파일을 만들고 **프라이머리** 사이트 설정의 마지막 단계에서 얻은 콘텐츠를 포함하세요:

   ```shell
   editor server.crt
   ```

1. **세컨더리** 사이트에서 PostgreSQL TLS 확인을 설정하세요:

   `server.crt` 파일을 설치하세요:

   ```shell
   install \
      -D \
      -o gitlab-psql \
      -g gitlab-psql \
      -m 0400 \
      -T server.crt ~gitlab-psql/.postgresql/root.crt
   ```

   PostgreSQL은 이제 TLS 연결을 확인할 때만 정확한 인증서를 인식합니다. 인증서는 프라이빗 키에 액세스할 수 있는 사용자만이 복제할 수 있으며, 이는 **프라이머리** 사이트에만 **only**.

1. `gitlab-psql` 사용자가 **프라이머리** 사이트의 데이터베이스에 연결할 수 있는지 테스트하세요(기본 데이터베이스 이름은 Linux 패키지 설치에서 `gitlabhq_production`입니다):

   ```shell
   sudo \
      -u gitlab-psql /opt/gitlab/embedded/bin/psql \
      --list \
      -U gitlab_replicator \
      -d "dbname=gitlabhq_production sslmode=verify-ca" \
      -W \
      -h <primary_site_ip>
   ```

   > [!note]
   > 수동으로 생성한 인증서를 사용하고 전체 호스트 이름 확인을 활용하기 위해 `sslmode=verify-full`를 사용하려면 명령을 실행할 때 `verify-ca`를 `verify-full`로 바꾸세요.

   메시지가 표시되면 첫 번째 단계에서 `gitlab_replicator` 사용자에 대해 설정한 평문 암호를 입력하세요. 모든 것이 제대로 작동했으면 **프라이머리** 사이트의 데이터베이스 목록을 확인할 수 있습니다.

   여기서 연결 실패는 TLS 구성이 올바르지 않음을 나타냅니다. **프라이머리** 사이트의 `~gitlab-psql/data/server.crt` 콘텐츠가 **세컨더리** 사이트의 `~gitlab-psql/.postgresql/root.crt` 콘텐츠와 일치하는지 확인하세요.

1. `/etc/gitlab/gitlab.rb`을 편집하고 역할을 `geo_secondary_role`로 설정하세요(자세한 내용은 [Geo 역할](https://docs.gitlab.com/omnibus/roles/#gitlab-geo-roles) 참조):

   ```ruby
   ##
   ## Geo Secondary role
   ## - configure dependent flags automatically to enable Geo
   ##
   roles(['geo_secondary_role'])
   ```

1. PostgreSQL을 구성하세요:

   이 단계는 **프라이머리** 인스턴스를 구성한 방식과 유사합니다. 단일 노드를 사용하는 경우에도 이를 활성화해야 합니다.

   > [!warning]
   > 각 암호 유형은 모든 Geo 사이트에서 [일치하는 값](#database-password-consistency-requirements)을 가져야 합니다.

   `/etc/gitlab/gitlab.rb`을 편집하고 IP 주소를 네트워크 구성에 적절한 주소로 바꾸면서 다음을 추가하세요:

   ```ruby
   ##
   ## Secondary address
   ## - replace '<secondary_site_ip>' with the public or VPC address of your Geo secondary site
   ##
   postgresql['listen_address'] = '<secondary_site_ip>'
   postgresql['md5_auth_cidr_addresses'] = ['<secondary_site_ip>/32']

   ##
   ## Database credentials password (defined previously in primary site)
   ## - replicate same values here as defined in primary site
   ##
   postgresql['sql_replication_password'] = '<md5_hash_of_your_replication_password>'
   postgresql['sql_user_password'] = '<md5_hash_of_your_db_password>'
   gitlab_rails['db_password'] = '<your_db_password_here>'
   ```

   외부 PostgreSQL 인스턴스의 경우 [추가 지침](external_database.md)을 참조하세요. 이전 **프라이머리** 사이트를 다시 온라인 상태로 가져와 **세컨더리** 사이트로 제공하려면 `roles(['geo_primary_role'])` 또는 `geo_primary_role['enable'] = true`를 제거해야 합니다.

1. GitLab을 다시 구성하여 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl reconfigure
   ```

1. IP 변경을 적용하기 위해 PostgreSQL을 다시 시작하세요:

   ```shell
   gitlab-ctl restart postgresql
   ```

#### 3단계. 복제 프로세스 시작 {#step-3-initiate-the-replication-process}

다음은 **세컨더리** 사이트의 데이터베이스를 **프라이머리** 사이트의 데이터베이스에 연결하는 스크립트입니다. 이 스크립트는 데이터베이스를 복제하고 스트리밍 복제에 필요한 파일을 만듭니다.

사용된 디렉터리는 Linux 패키지 설치에서 설정되는 기본값입니다. 기본값을 변경한 경우 스크립트를 그에 따라 구성하세요(디렉터리 및 경로 바꾸기).

> [!warning]
> **세컨더리** 사이트에서 이를 실행하도록 주의하세요. `pg_basebackup`를 실행하기 전에 모든 PostgreSQL 데이터를 제거합니다.

1. GitLab **세컨더리** 사이트로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. [데이터베이스 친화적인 이름](https://www.postgresql.org/docs/16/warm-standby.html#STREAMING-REPLICATION-SLOTS-MANIPULATION)을 선택하여 **세컨더리** 사이트에 대해 복제 슬롯 이름으로 사용할 수 있도록 하세요. 예를 들어 도메인이 `secondary.geo.example.com`이면 아래 명령에 표시된 대로 슬롯 이름으로 `secondary_example`를 사용하세요.

1. 아래 명령을 실행하여 백업/복원을 시작하고 복제를 시작하세요

   > [!warning]
   > 각 Geo **세컨더리** 사이트는 자신의 고유한 복제 슬롯 이름을 가져야 합니다. 두 보조 사이트 간에 동일한 슬롯 이름을 사용하면 PostgreSQL 복제가 중단됩니다.

   복제 슬롯 이름은 소문자, 숫자 및 밑줄 문자만 포함해야 합니다.

   메시지가 표시되면 첫 번째 단계에서 `gitlab_replicator` 사용자에 대해 설정한 평문 암호를 입력하세요.

   ```shell
   gitlab-ctl replicate-geo-database \
      --slot-name=<secondary_site_name> \
      --host=<primary_site_ip> \
      --sslmode=verify-ca
   ```

   > [!note]
   > 사용자 지정 PostgreSQL 인증서를 생성한 경우 `--sslmode=verify-full`를 사용해야 합니다(또는 `sslmode` 줄을 완전히 생략). 추가 보안을 위해 인증서 CN/SAN의 전체 호스트 이름에 대한 추가 검증을 활용합니다. 그 대신 자동으로 생성된 인증서를 `verify-full`와 함께 사용하면 `--host` 값과 일치하지 않는 일반적인 `PostgreSQL` CN이 있기 때문에 실패합니다.

   이 명령은 많은 추가 옵션을 사용합니다. `--help`을 사용하여 모두 나열할 수 있지만 몇 가지 팁은 다음과 같습니다:

   - 프라이머리 사이트에 단일 노드가 있으면 `--host` 매개변수로 프라이머리 노드 호스트를 사용하세요.
   - 프라이머리 사이트가 외부 PostgreSQL 데이터베이스를 사용하는 경우 `--host` 매개변수를 조정해야 합니다:
     - PgBouncer 설정의 경우 실제 PostgreSQL 데이터베이스 호스트를 직접 대상으로 지정하고 PgBouncer 주소는 아닙니다.
     - Patroni 구성의 경우 현재 Patroni 리더 호스트를 대상으로 지정하세요.
     - 로드 밸런서를 사용할 때(예: HAProxy), 로드 밸런서가 항상 Patroni 리더로 라우팅하도록 구성된 경우 로드 밸런서를 대상으로 지정할 수 있습니다. 그렇지 않으면 실제 데이터베이스 호스트를 대상으로 지정해야 합니다.
     - 전용 PostgreSQL 노드가 있는 설정의 경우 전용 데이터베이스 호스트를 직접 대상으로 지정하세요.
   - `--slot-name`를 **프라이머리** 데이터베이스에서 사용할 복제 슬롯의 이름으로 변경하세요. 스크립트는 존재하지 않는 경우 복제 슬롯을 자동으로 만들려고 시도합니다.
   - PostgreSQL이 표준이 아닌 포트에서 수신 대기하는 경우 `--port=`를 추가하세요.
   - 데이터베이스가 너무 커서 30분 내에 전송할 수 없는 경우 시간 초과를 늘려야 합니다. 예를 들어 초기 복제가 1시간 이내에 완료될 것으로 예상되면 `--backup-timeout=3600`를 사용하세요.
   - `--sslmode=disable`을 전달하여 PostgreSQL TLS 인증을 완전히 건너뛰세요(예를 들어 네트워크 경로가 안전하다는 것을 알고 있거나 사이트 간 VPN을 사용하고 있는 경우). 공용 인터넷을 통해서는 **not**!
   - [PostgreSQL 문서](https://www.postgresql.org/docs/16/libpq-ssl.html#LIBPQ-SSL-PROTECTION)에서 각 `sslmode`에 대한 자세한 내용을 확인할 수 있습니다. 이전에 나열된 지침은 수동적인 도청자와 능동형 "중간자" 공격자 모두로부터 보호를 보장하기 위해 신중하게 작성되었습니다.
   - 구식 사이트를 Geo **세컨더리** 사이트로 변경하려면 명령줄에 `--force`를 추가해야 합니다.
   - 비프로덕션 머신에서는 `--skip-backup`를 추가하여 백업 단계를 비활성화할 수 있습니다(이것이 원하는 것임을 확실한 경우).

복제 프로세스가 이제 완료되었습니다.

> [!note]
> 복제 프로세스는 프라이머리 사이트의 데이터베이스에서 세컨더리 사이트의 데이터베이스로만 데이터를 복사합니다. 세컨더리 사이트 구성을 완료하려면 [프라이머리 사이트에 세컨더리 사이트를 추가](../replication/configuration.md#step-3-add-the-secondary-site)하세요.

### PgBouncer 지원(선택 사항) {#pgbouncer-support-optional}

[PgBouncer](https://www.pgbouncer.org/)를 GitLab Geo와 함께 사용하여 PostgreSQL 연결을 풀링할 수 있으며, 단일 인스턴스 설치를 사용할 때도 성능을 향상시킬 수 있습니다.

Geo **프라이머리** 사이트를 지원하는 노드 클러스터와 Geo **세컨더리** 사이트를 지원하는 다른 두 클러스터를 포함한 고가용성 구성에서 GitLab을 사용하는 경우 PgBouncer를 사용해야 합니다. 주 데이터베이스용과 추적 데이터베이스용의 두 개의 PgBouncer 노드가 필요합니다. 자세한 내용은 [관련 문서](../../postgresql/replication_and_failover.md)를 참조하세요.

### 복제 암호 변경 {#changing-the-replication-password}

> [!warning]
> 복제 암호를 변경할 때는 **전체** Geo 사이트(프라이머리 및 모든 세컨더리)에서 [동일한 암호 값](#database-password-consistency-requirements)으로 업데이트해야 합니다. 암호를 동기화된 상태로 유지하지 못하면 복제가 중단됩니다.

Linux 패키지 설치로 관리되는 PostgreSQL 인스턴스를 사용할 때 [복제 사용자](https://www.postgresql.org/docs/16/warm-standby.html#STREAMING-REPLICATION)의 암호를 변경하려면:

GitLab Geo **프라이머리** 사이트에서:

1. 복제 사용자의 기본값은 `gitlab_replicator`이지만 `/etc/gitlab/gitlab.rb`에서 `postgresql['sql_replication_user']` 설정 아래에 사용자 정의 복제 사용자를 설정한 경우 사용자 정의 사용자에 맞게 다음 지침을 조정하세요.

   원하는 비밀번호의 MD5 해시를 생성합니다:

   ```shell
   sudo gitlab-ctl pg-password-md5 gitlab_replicator
   # Enter password: <your_replication_password_here>
   # Confirm password: <your_replication_password_here>
   # 950233c0dfc2f39c64cf30457c3b7f1e
   ```

   `/etc/gitlab/gitlab.rb`을(를) 편집합니다:

   ```ruby
   # Fill with the hash generated by `gitlab-ctl pg-password-md5 gitlab_replicator`
   postgresql['sql_replication_password'] = '<md5_hash_of_your_replication_password>'
   ```

1. 파일을 저장하고 GitLab을 다시 구성하여 PostgreSQL에서 복제 사용자의 암호를 변경하세요:

   ```shell
   sudo gitlab-ctl reconfigure
   ```

1. 복제 암호 변경을 적용하기 위해 PostgreSQL을 다시 시작하세요:

   ```shell
   sudo gitlab-ctl restart postgresql
   ```

암호가 모든 **세컨더리** 사이트에서 업데이트될 때까지 세컨더리의 [PostgreSQL 로그](../../logs/_index.md#postgresql-logs)에서 다음 오류 메시지를 보고합니다:

```console
FATAL:  could not connect to the primary server: FATAL:  password authentication failed for user "gitlab_replicator"
```

모든 GitLab Geo **세컨더리** 사이트에서:

1. 첫 번째 단계는 구성 관점에서 필요하지 않습니다. 해시된 `'sql_replication_password'`이 GitLab Geo **세컨더리** 사이트에서 사용되지 않기 때문입니다. 그러나 **세컨더리** 사이트를 GitLab Geo **프라이머리**로 승격해야 하는 경우 **세컨더리** 사이트 구성에서 `'sql_replication_password'`를 일치하는지 확인하세요.

   `/etc/gitlab/gitlab.rb`을(를) 편집합니다:

   ```ruby
   # Fill with the hash generated by `gitlab-ctl pg-password-md5 gitlab_replicator` on the Geo primary
   postgresql['sql_replication_password'] = '<md5_hash_of_your_replication_password>'
   ```

1. 초기 복제 설정 중에 `gitlab-ctl replicate-geo-database` 명령은 복제 사용자 계정의 평문 암호를 두 위치에 씁니다:

   - `gitlab-geo.conf`:  PostgreSQL 복제 프로세스에서 사용하고 PostgreSQL 데이터 디렉터리(기본적으로 `/var/opt/gitlab/postgresql/data/gitlab-geo.conf`)에 기록합니다.
   - `.pgpass`:  `gitlab-psql` 사용자가 사용하며, 기본적으로 `/var/opt/gitlab/postgresql/.pgpass`에 있습니다.

   두 파일의 평문 암호를 업데이트하고 PostgreSQL을 다시 시작하세요:

   ```shell
   sudo gitlab-ctl restart postgresql
   ```

## 다중 노드 데이터베이스 복제 {#multi-node-database-replication}

### 단일 PostgreSQL 노드를 Patroni로 마이그레이션 {#migrating-a-single-postgresql-node-to-patroni}

Patroni 도입 전에는 Geo가 **세컨더리** 사이트의 HA 설정을 위해 Linux 패키지 설치를 지원하지 않았습니다.

Patroni를 사용하면 이제 이러한 지원이 가능합니다. 기존 PostgreSQL을 Patroni로 마이그레이션하려면:

1. **프라이머리** 사이트에 설정한 방식과 유사하게 세컨더리에서 Consul 클러스터 설정을 한 후를 확인하세요.
1. [영구적인 복제 슬롯 구성](#step-1-configure-patroni-permanent-replication-slot-on-the-primary-site)하세요.
1. [내부 로드 밸런서 구성](#step-2-configure-the-internal-load-balancer-on-the-primary-site)하세요.
1. [PgBouncer 노드 구성](#step-3-configure-pgbouncer-nodes-on-the-secondary-site)하세요
1. [스탠바이 클러스터 구성](#step-4-configure-a-standby-cluster-on-the-secondary-site)을 해당 단일 노드 머신에서 실행하세요.

최종적으로 단일 노드가 있는 **Standby Cluster**를 갖게 됩니다. 이를 통해 이전에 나열된 동일한 지침에 따라 추가 Patroni 노드를 추가할 수 있습니다.

### Patroni 지원 {#patroni-support}

Patroni는 Geo의 공식 복제 관리 솔루션입니다. Patroni는 **프라이머리** 및 **세컨더리** Geo 사이트에서 고가용성 클러스터를 구축하는 데 사용할 수 있습니다. **세컨더리** 사이트에서 Patroni를 사용하는 것은 선택적이며 각 Geo 사이트에서 동일한 수의 노드를 사용할 필요가 없습니다.

프라이머리 사이트에서 Patroni를 설정하는 방법에 대한 지침은 [관련 문서](../../postgresql/replication_and_failover.md#patroni)를 참조하세요.

#### Geo 세컨더리 사이트에 대한 Patroni 클러스터 구성 {#configuring-patroni-cluster-for-a-geo-secondary-site}

Geo 세컨더리 사이트에서 주 PostgreSQL 데이터베이스는 프라이머리 사이트의 PostgreSQL 데이터베이스의 읽기 전용 복제본입니다.

프로덕션 준비 및 보안 설정에는 최소한 다음이 필요합니다:

- 3개의 Consul 노드 _(프라이머리 및 세컨더리 사이트)_
- 2개의 Patroni 노드 _(프라이머리 및 세컨더리 사이트)_
- 1개의 PgBouncer 노드 _(프라이머리 및 세컨더리 사이트)_
- 1개의 내부 로드 밸런서 _(프라이머리 사이트만)_

내부 로드 밸런서는 새 리더가 선출될 때마다 Patroni 클러스터의 리더에 연결하기 위한 단일 엔드포인트를 제공합니다. 로드 밸런서는 세컨더리 사이트에서 계단식 복제를 활성화하는 데 필요합니다.

[암호 자격 증명](../../postgresql/replication_and_failover.md#database-authorization-for-patroni)과 기타 데이터베이스 모범 사례를 사용해야 합니다.

##### 1단계. 프라이머리 사이트에서 Patroni 영구적인 복제 슬롯 구성 {#step-1-configure-patroni-permanent-replication-slot-on-the-primary-site}

프라이머리 데이터베이스에서 Patroni 클러스터의 세컨더리 노드로 연속적인 데이터 복제를 보장하기 위해 프라이머리 데이터베이스에서 영구적인 복제 슬롯을 설정하세요.

세컨더리 사이트에서 Patroni를 사용한 데이터베이스 복제를 설정하려면 프라이머리 사이트의 Patroni 클러스터에서 영구적인 복제 슬롯을 구성해야 하며 암호 인증이 사용되는지 확인해야 합니다.

프라이머리 사이트에서 Patroni 인스턴스를 실행하는 각 노드에서 **starting on the Patroni Leader instance**:

1. Patroni 인스턴스로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. GitLab을 업그레이드할 때 예기치 않은 다운타임을 피하기 위해 [자동 PostgreSQL 업그레이드를 거부합니다](https://docs.gitlab.com/omnibus/settings/database/#opt-out-of-automatic-postgresql-upgrades). Geo를 사용한 PostgreSQL 업그레이드에 관한 알려진 [주의사항](https://docs.gitlab.com/omnibus/settings/database/#caveats-when-upgrading-postgresql-with-geo)을 인식하세요. 특히 더 큰 환경의 경우 PostgreSQL 업그레이드를 신중하게 계획하고 실행해야 합니다. 그 결과 향후로 PostgreSQL 업그레이드가 정기적인 유지보수 활동의 일부가 되도록 합니다.
1. `/etc/gitlab/gitlab.rb`을 편집하고 다음을 추가하세요. 각 암호 유형이 모든 Geo 사이트에서 [일치하는 값](#database-password-consistency-requirements)을 가지고 있는지 확인하세요.

   ```ruby
   roles(['patroni_role'])

   consul['services'] = %w(postgresql)
   consul['configuration'] = {
     retry_join: %w[CONSUL_PRIMARY1_IP CONSUL_PRIMARY2_IP CONSUL_PRIMARY3_IP]
   }

   # You need one entry for each secondary, with a unique name following PostgreSQL slot_name constraints:
   #
   # Configuration syntax is: 'unique_slotname' => { 'type' => 'physical' },
   # We don't support setting a permanent replication slot for logical replication type
   patroni['replication_slots'] = {
     'geo_secondary' => { 'type' => 'physical' }
   }

   patroni['use_pg_rewind'] = true
   patroni['postgresql']['max_wal_senders'] = 8 # Use double of the amount of patroni/reserved slots (3 patronis + 1 reserved slot for a Geo secondary).
   patroni['postgresql']['max_replication_slots'] = 8 # Use double of the amount of patroni/reserved slots (3 patronis + 1 reserved slot for a Geo secondary).
   patroni['username'] = 'PATRONI_API_USERNAME'
   patroni['password'] = 'PATRONI_API_PASSWORD'
   patroni['replication_password'] = 'PLAIN_TEXT_POSTGRESQL_REPLICATION_PASSWORD'

   # Add all patroni nodes to the allowlist
   patroni['allowlist'] = %w[
     127.0.0.1/32
     PATRONI_PRIMARY1_IP/32 PATRONI_PRIMARY2_IP/32 PATRONI_PRIMARY3_IP/32
     PATRONI_SECONDARY1_IP/32 PATRONI_SECONDARY2_IP/32 PATRONI_SECONDARY3_IP/32
   ]

   # We list all secondary instances as they can all become a Standby Leader
   postgresql['md5_auth_cidr_addresses'] = %w[
     PATRONI_PRIMARY1_IP/32 PATRONI_PRIMARY2_IP/32 PATRONI_PRIMARY3_IP/32 PATRONI_PRIMARY_PGBOUNCER/32
     PATRONI_SECONDARY1_IP/32 PATRONI_SECONDARY2_IP/32 PATRONI_SECONDARY3_IP/32 PATRONI_SECONDARY_PGBOUNCER/32
   ]

   postgresql['pgbouncer_user_password'] = 'PGBOUNCER_PASSWORD_HASH'
   postgresql['sql_replication_password'] = 'POSTGRESQL_REPLICATION_PASSWORD_HASH'
   postgresql['sql_user_password'] = 'POSTGRESQL_PASSWORD_HASH'
   postgresql['listen_address'] = '0.0.0.0' # You can use a public or VPC address here instead
   ```

1. GitLab을 다시 구성하여 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl reconfigure
   ```

   ```sql
   select pg_create_physical_replication_slot('geo_secondary')
   ```

1. 선택사항. 프라이머리에 PgBouncer가 없지만 세컨더리에 있는 경우:

   프라이머리 사이트에서 `pgbouncer` 사용자를 구성하고 Linux 패키지에 포함된 `pg_shadow_lookup` 함수를 추가하세요. 세컨더리 서버의 PgBouncer는 여전히 세컨더리 사이트의 PostgreSQL 노드에 연결할 수 있어야 합니다.

   ```sql
   --- Create a new user 'pgbouncer'
   CREATE USER pgbouncer;

   --- Set/change a password and grants replication privilege
   ALTER USER pgbouncer WITH REPLICATION ENCRYPTED PASSWORD '<pgbouncer_password_from_secondary>';

   CREATE OR REPLACE FUNCTION public.pg_shadow_lookup(in i_username text, out username text, out password text) RETURNS record AS $$
   BEGIN
       SELECT usename, passwd FROM pg_catalog.pg_shadow
       WHERE usename = i_username INTO username, password;
       RETURN;
   END;
   $$ LANGUAGE plpgsql SECURITY DEFINER;

   REVOKE ALL ON FUNCTION public.pg_shadow_lookup(text) FROM public, pgbouncer;
   GRANT EXECUTE ON FUNCTION public.pg_shadow_lookup(text) TO pgbouncer;
   ```

##### 2단계. 프라이머리 사이트에서 내부 로드 밸런서 구성 {#step-2-configure-the-internal-load-balancer-on-the-primary-site}

프라이머리 사이트에서 새 리더가 선출될 때마다 세컨더리 사이트의 스탠바이 리더를 다시 구성하는 것을 피하려면 TCP 내부 로드 밸런서를 설정해야 합니다. 이 로드 밸런서는 Patroni 클러스터의 리더에 연결하기 위한 단일 엔드포인트를 제공합니다.

Linux 패키지는 로드 밸런서를 포함하지 않습니다. [HAProxy](https://www.haproxy.org/)를 사용하여 이를 수행하는 방법은 다음과 같습니다.

다음 IP와 이름이 예로 사용됩니다:

- `10.6.0.21`:  Patroni 1 (`patroni1.internal`)
- `10.6.0.22`:  Patroni 2 (`patroni2.internal`)
- `10.6.0.23`:  Patroni 3 (`patroni3.internal`)

```plaintext
global
    log /dev/log local0
    log localhost local1 notice
    log stdout format raw local0

defaults
    log global
    default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions

frontend internal-postgresql-tcp-in
    bind *:5432
    mode tcp
    option tcplog

    default_backend postgresql

backend postgresql
    mode tcp
    option httpchk
    http-check expect status 200

    server patroni1.internal 10.6.0.21:5432 maxconn 100 check port 8008
    server patroni2.internal 10.6.0.22:5432 maxconn 100 check port 8008
    server patroni3.internal 10.6.0.23:5432 maxconn 100 check port 8008
```

자세한 지침은 선호하는 로드 밸런서의 문서를 참조하세요.

##### 3단계. 세컨더리 사이트에서 PgBouncer 노드 구성 {#step-3-configure-pgbouncer-nodes-on-the-secondary-site}

프로덕션 준비 및 고가용성 구성에는 최소한 3개의 Consul 노드와 최소 1개의 PgBouncer 노드가 필요합니다. 그러나 데이터베이스 노드당 하나의 PgBouncer 노드를 두는 것이 좋습니다. 1개 이상의 PgBouncer 서비스 노드가 있을 때 내부 로드 밸런서(TCP)가 필요합니다. 내부 로드 밸런서는 PgBouncer 클러스터에 연결하기 위한 단일 엔드포인트를 제공합니다. 자세한 내용은 [관련 문서](../../postgresql/replication_and_failover.md)를 참조하세요.

**세컨더리** 사이트에서 PgBouncer 인스턴스를 실행하는 각 노드에서:

1. PgBouncer 노드로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. `/etc/gitlab/gitlab.rb`을 편집하고 다음을 추가하세요:

   ```ruby
   # Disable all components except Pgbouncer and Consul agent
   roles(['pgbouncer_role'])

   # PgBouncer configuration
   pgbouncer['admin_users'] = %w(pgbouncer gitlab-consul)
   pgbouncer['users'] = {
   'gitlab-consul': {
      # Generate it with: `gitlab-ctl pg-password-md5 gitlab-consul`
      password: 'GITLAB_CONSUL_PASSWORD_HASH'
    },
     'pgbouncer': {
       # Generate it with: `gitlab-ctl pg-password-md5 pgbouncer`
       password: 'PGBOUNCER_PASSWORD_HASH'
     }
   }

   # Consul configuration
   consul['watchers'] = %w(postgresql)
   consul['configuration'] = {
     retry_join: %w[CONSUL_SECONDARY1_IP CONSUL_SECONDARY2_IP CONSUL_SECONDARY3_IP]
   }
   consul['monitoring_service_discovery'] =  true
   ```

1. GitLab을 다시 구성하여 변경 사항이 적용되도록 하세요:

   ```shell
   gitlab-ctl reconfigure
   ```

1. Consul이 PgBouncer를 다시 로드할 수 있도록 `.pgpass` 파일을 만드세요. 메시지가 표시되면 `PLAIN_TEXT_PGBOUNCER_PASSWORD`를 두 번 입력하세요:

   ```shell
   gitlab-ctl write-pgpass --host 127.0.0.1 --database pgbouncer --user pgbouncer --hostuser gitlab-consul
   ```

1. PgBouncer 서비스를 다시 로드하세요:

   ```shell
   gitlab-ctl hup pgbouncer
   ```

##### 4단계. 세컨더리 사이트에서 스탠바이 클러스터 구성 {#step-4-configure-a-standby-cluster-on-the-secondary-site}

> [!note]
> 단일 PostgreSQL 인스턴스가 있는 세컨더리 사이트를 Patroni 클러스터로 변환하는 경우 PostgreSQL 인스턴스에서 시작해야 합니다. Patroni 스탠바이 리더 인스턴스가 되고 필요하면 다른 복제본으로 전환할 수 있습니다.

세컨더리 사이트에서 Patroni 인스턴스를 실행하는 각 노드에 대해:

1. Patroni 노드로 SSH하여 루트로 로그인하세요:

   ```shell
   sudo -i
   ```

1. GitLab을 업그레이드할 때 예기치 않은 다운타임을 피하기 위해 [자동 PostgreSQL 업그레이드를 거부합니다](https://docs.gitlab.com/omnibus/settings/database/#opt-out-of-automatic-postgresql-upgrades). Geo를 사용한 PostgreSQL 업그레이드에 관한 알려진 [주의사항](https://docs.gitlab.com/omnibus/settings/database/#caveats-when-upgrading-postgresql-with-geo)을 인식하세요. 특히 더 큰 환경의 경우 PostgreSQL 업그레이드를 신중하게 계획하고 실행해야 합니다. 그 결과 향후로 PostgreSQL 업그레이드가 정기적인 유지보수 활동의 일부가 되도록 합니다.
1. `/etc/gitlab/gitlab.rb`을 편집하고 다음을 추가하세요:

   > [!warning]
   > 각 암호 유형은 모든 Geo 사이트에서 [일치하는 값](#database-password-consistency-requirements)을 가져야 합니다.

   ```ruby
   roles(['consul_role', 'patroni_role'])

   consul['enable'] = true
   consul['configuration'] = {
     retry_join: %w[CONSUL_SECONDARY1_IP CONSUL_SECONDARY2_IP CONSUL_SECONDARY3_IP]
   }
   consul['services'] = %w(postgresql)

   postgresql['md5_auth_cidr_addresses'] = [
     'PATRONI_SECONDARY1_IP/32', 'PATRONI_SECONDARY2_IP/32', 'PATRONI_SECONDARY3_IP/32', 'PATRONI_SECONDARY_PGBOUNCER/32',
     # Any other instance that needs access to the database as per documentation
   ]


   # Add patroni nodes to the allowlist
   patroni['allowlist'] = %w[
     127.0.0.1/32
     PATRONI_SECONDARY1_IP/32 PATRONI_SECONDARY2_IP/32 PATRONI_SECONDARY3_IP/32
   ]

   patroni['standby_cluster']['enable'] = true
   patroni['standby_cluster']['host'] = 'INTERNAL_LOAD_BALANCER_PRIMARY_IP'
   patroni['standby_cluster']['port'] = INTERNAL_LOAD_BALANCER_PRIMARY_PORT
   patroni['standby_cluster']['primary_slot_name'] = 'geo_secondary' # Or the unique replication slot name you setup before
   patroni['username'] = 'PATRONI_API_USERNAME'
   patroni['password'] = 'PATRONI_API_PASSWORD'
   patroni['replication_password'] = 'PLAIN_TEXT_POSTGRESQL_REPLICATION_PASSWORD'
   patroni['use_pg_rewind'] = true
   patroni['postgresql']['max_wal_senders'] = 5 # A minimum of three for one replica, plus two for each additional replica
   patroni['postgresql']['max_replication_slots'] = 5 # A minimum of three for one replica, plus two for each additional replica

   postgresql['pgbouncer_user_password'] = 'PGBOUNCER_PASSWORD_HASH'
   postgresql['sql_replication_password'] = 'POSTGRESQL_REPLICATION_PASSWORD_HASH'
   postgresql['sql_user_password'] = 'POSTGRESQL_PASSWORD_HASH'
   postgresql['listen_address'] = '0.0.0.0' # You can use a public or VPC address here instead

   # GitLab Rails configuration is required for `gitlab-ctl geo-replication-pause`
   gitlab_rails['db_password'] = 'POSTGRESQL_PASSWORD'
   gitlab_rails['enable'] = true
   gitlab_rails['auto_migrate'] = false
   ```

   `patroni['standby_cluster']['host']` 및 `patroni['standby_cluster']['port']`를 구성할 때:
   - `INTERNAL_LOAD_BALANCER_PRIMARY_IP`은 프라이머리 내부 로드 밸런서 IP를 가리켜야 합니다.
   - `INTERNAL_LOAD_BALANCER_PRIMARY_PORT`은 [프라이머리 Patroni 클러스터 리더에 대해 구성된](#step-2-configure-the-internal-load-balancer-on-the-primary-site) 프론트엔드 포트를 가리켜야 합니다. PgBouncer 프론트엔드 포트를 **Do not**.

1. GitLab을 다시 구성하여 변경 사항이 적용되도록 하세요. 이 단계는 PostgreSQL 사용자 및 설정을 부트스트랩하는 데 필요합니다.

   - 이것이 Patroni의 새로운 설치인 경우:

     ```shell
     gitlab-ctl reconfigure
     ```

   - 이전에 작동하는 Patroni 클러스터가 있던 사이트에서 Patroni 스탠바이 클러스터를 구성하는 경우:

     1. Patroni에서 관리하는 모든 노드(계단식 복제본 포함)에서 Patroni를 중지하세요:

        ```shell
        gitlab-ctl stop patroni
        ```

     1. 스탠바이 클러스터를 다시 만들기 위해 리더 Patroni 노드에서 다음을 실행하세요:

        ```shell
        rm -rf /var/opt/gitlab/postgresql/data
        /opt/gitlab/embedded/bin/patronictl -c /var/opt/gitlab/patroni/patroni.yaml remove postgresql-ha
        gitlab-ctl reconfigure
        ```

     1. 리더 Patroni 노드에서 Patroni를 시작하여 프라이머리 데이터베이스에서 복제 프로세스를 시작하세요:

        ```shell
        gitlab-ctl start patroni
        ```

     1. Patroni 클러스터의 상태를 확인하세요:

        ```shell
        gitlab-ctl patroni members
        ```

        다음을 확인하세요:

        - 현재 Patroni 노드가 출력에 나타납니다.
        - 역할은 `Standby Leader`입니다. 역할은 처음에 `Replica`를 표시할 수 있습니다.
        - 상태는 `Running`입니다. 상태는 처음에 `Creating replica`를 표시할 수 있습니다.

        노드의 역할이 `Standby Leader`로 안정화되고 상태가 `Running`가 될 때까지 기다리세요. 이는 몇 분 정도 걸릴 수 있습니다.

     1. 리더 Patroni 노드가 `Standby Leader`이고 `Running`일 때 스탠바이 클러스터의 다른 Patroni 노드에서 Patroni를 시작하세요:

        ```shell
        gitlab-ctl start patroni
        ```

        다른 Patroni 노드는 새 스탠바이 클러스터에 복제본으로 참여하고 리더 Patroni 노드에서 자동으로 복제를 시작합니다.

1. 클러스터 상태를 확인하세요:

   ```shell
   gitlab-ctl patroni members
   ```

   모든 Patroni 노드가 `Running` 상태에 나열되는지 확인하세요. 1개의 `Standby Leader` 노드와 여러 `Replica` 노드가 있어야 합니다.

### 단일 추적 데이터베이스 노드를 Patroni로 마이그레이션 {#migrating-a-single-tracking-database-node-to-patroni}

Patroni 도입 전에는 Geo가 세컨더리 사이트의 HA 설정을 위해 Linux 패키지 설치를 지원하지 않았습니다.

Patroni를 사용하면 이제 HA 설정을 지원할 수 있습니다. 그러나 Patroni의 일부 제한 사항으로 인해 동일한 머신에서 두 개의 서로 다른 클러스터를 관리할 수 없습니다. [Geo 세컨더리 사이트를 위한 Patroni 클러스터를 구성하는 방법](#configuring-patroni-cluster-for-a-geo-secondary-site)을 설명하는 동일한 지침에 따라 추적 데이터베이스를 위한 새로운 Patroni 클러스터를 설정해야 합니다.

세컨더리 노드는 새 추적 데이터베이스를 백필하며 데이터 동기화가 필요하지 않습니다.

### 추적 PostgreSQL 데이터베이스에 대한 Patroni 클러스터 구성 {#configuring-patroni-cluster-for-the-tracking-postgresql-database}

**세컨더리** Geo 사이트는 별도의 PostgreSQL 설치를 추적 데이터베이스로 사용하여 복제 상태를 추적하고 잠재적인 복제 문제로부터 자동으로 복구합니다.

Geo 추적 데이터베이스를 단일 노드에서 실행하려면 [Geo 세컨더리 사이트에서 Geo 추적 데이터베이스 구성](../replication/multiple_servers.md#step-2-configure-the-geo-tracking-database-on-the-geo-secondary-site)을 참조하세요.

Linux 패키지는 고가용성 구성에서 Geo 추적 데이터베이스를 실행하는 것을 지원하지 않습니다. 특히 장애 조치가 제대로 작동하지 않습니다. [기능 요청 문제](https://gitlab.com/gitlab-org/omnibus-gitlab/-/issues/7292)를 참조하세요.

Geo 추적 데이터베이스를 고가용성 구성에서 실행하려면 세컨더리 사이트를 클라우드 관리 데이터베이스와 같은 외부 PostgreSQL 데이터베이스 또는 수동으로 구성된 [Patroni](https://patroni.readthedocs.io/en/latest/) 클러스터(GitLab Linux 패키지로 관리되지 않음)에 연결할 수 있습니다. [외부 PostgreSQL 인스턴스를 사용한 Geo](external_database.md#configure-the-tracking-database)를 따르세요.
