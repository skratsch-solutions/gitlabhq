GitLab Geo 사이트의 다양한 설정을 구성할 수 있습니다. 자세한 내용은 [Geo 문서](geo/_index.md)를 참조하세요.

전제 조건:

- 관리자 액세스 권한이 있어야 합니다.

프라이머리 사이트 또는 세컨더리 사이트에서:

1. 오른쪽 위 모서리에서 **Admin**을 선택합니다.
1. 왼쪽 사이드바에서 **Geo** > **사이트**를 선택합니다.

## 공통 설정 {#common-settings}

모든 Geo 사이트에는 다음 설정이 있습니다:

| 설정 | 설명 |
| --------| ----------- |
| 프라이머리 | 이것은 Geo 사이트를 **프라이머리** 사이트로 표시합니다. **프라이머리** 사이트는 하나만 존재할 수 있습니다. |
| 이름    | Geo 사이트의 고유 식별자입니다. 물리적 위치를 이름으로 사용하는 것이 매우 권장됩니다. 좋은 예는 `London Office` 또는 `us-east-1`입니다. `primary`, `secondary`, `Geo` 또는 `DR`과 같은 단어는 피하세요. 이렇게 하면 물리적 위치는 변하지 않지만 Geo 사이트 역할은 변할 수 있으므로 장애 조치 프로세스가 더 쉬워집니다. 단일 Geo 사이트의 모든 노드는 동일한 사이트 이름을 사용합니다. 노드는 `gitlab_rails['geo_node_name']` 설정을 `/etc/gitlab/gitlab.rb`에서 사용하여 PostgreSQL 데이터베이스의 Geo 사이트 기록을 조회합니다. `gitlab_rails['geo_node_name']`가 설정되지 않으면 노드의 `external_url` (뒤에 슬래시 포함)가 대체값으로 사용됩니다. `Name`의 값은 대소문자를 구분하며 대부분의 문자가 허용됩니다. |
| URL     | 인스턴스의 사용자 마주보는 URL입니다. |

### 허용된 Geo IP {#allowed-geo-ip}

**허용된 Geo IP** 설정은 세컨더리 사이트에서 프라이머리 사이트로 요청을 할 수 있는 IP 주소를 제어합니다. 프라이머리 사이트는 이 설정을 사용하여 다음을 검증합니다:

- 세컨더리 사이트의 Git HTTP 요청입니다.
- 세컨더리 사이트의 Geo API 요청입니다.

**허용된 Geo IP** 설정:

- 세컨더리 사이트에는 영향을 주지 않습니다. 이 설정은 데이터베이스의 세컨더리 사이트로 복제되지만 해당 위치에서 사용되지 않습니다.
- 쉼표로 구분된 IP 주소 및 CIDR 블록 목록을 허용합니다(예: `192.168.1.1, 10.0.0.0/8, 2001:db8::/32`).
- 기본값은 `0.0.0.0/0, ::/0`이며, 이는 모든 IP 주소에서의 요청을 허용합니다.
- 데이터베이스가 읽기 전용이므로 세컨더리 사이트에서는 수정할 수 없습니다.

## 세컨더리 사이트 설정 {#secondary-site-settings}

**세컨더리** 사이트에는 여러 추가 설정이 있습니다:

| 설정                   | 설명 |
|---------------------------|-------------|
| 선택적 동기화 | 이 **세컨더리** 사이트에 대해 Geo [선택적 동기화](geo/replication/selective_synchronization.md)를 활성화합니다. |
| 리포지토리 동기화 용량  | 이 **세컨더리** 사이트가 **프라이머리** 사이트에 대해 리포지토리를 백필할 때 수행하는 동시 요청 수입니다. |
| 파일 동기화 용량        | 이 **세컨더리** 사이트가 **프라이머리** 사이트에 대해 파일을 백필할 때 수행하는 동시 요청 수입니다. |

## Geo 백필 {#geo-backfill}

**세컨더리** 사이트는 **프라이머리** 사이트에서 리포지토리 및 파일의 변경 사항이 있다는 알림을 받으며, 항상 이러한 변경 사항을 가능한 한 빨리 동기화하려고 시도합니다.

백필은 **세컨더리** 사이트가 데이터베이스에 추가되기 전에 존재했던 리포지토리 및 파일로 **세컨더리** 사이트를 채우는 행위입니다. 리포지토리 및 파일의 수가 매우 많을 수 있으므로 모두를 한 번에 다운로드하려고 시도하는 것은 불가능합니다. 따라서 GitLab은 이러한 작업의 동시성에 상한선을 설정합니다.

백필이 소요되는 시간은 최대 동시성에 따라 달라지지만, 값이 높을수록 **프라이머리** 사이트에 더 많은 부담을 줍니다. 제한은 구성 가능합니다. **프라이머리** 사이트의 여유 용량이 많으면 값을 증가시켜 더 짧은 시간 내에 백필을 완료할 수 있습니다. 많은 부하가 걸려 있고 백필이 표준 요청의 가용성을 줄이면 값을 줄일 수 있습니다.

## 내부 URL 설정 {#set-up-the-internal-urls}

프라이머리와 세컨더리 사이트 간의 동기화를 위해 다른 URL을 설정할 수 있습니다.

**프라이머리** 사이트의 내부 URL은 **세컨더리** 사이트에서 이를 연결하는 데 사용됩니다. 예를 들어, 리포지토리를 동기화하기 위해 사용합니다. 내부 URL이라는 이름은 사용자가 사용하는 [외부 URL](https://docs.gitlab.com/omnibus/settings/configuration/#configure-the-external-url-for-gitlab)과 구별됩니다. 내부 URL은 비공개 주소일 필요가 없습니다.

**세컨더리** 사이트의 내부 URL은 **프라이머리** 사이트에서 이를 연결하는 데 사용됩니다. 예를 들어, **Geo** > **사이트** > **프로젝트 리포지토리**의 운영자 영역에 표시하기 위해 동기화 또는 확인 추적 메타데이터를 검색하는 데 사용됩니다.

내부 URL은 기본값으로 외부 URL로 설정됩니다. 변경하려면:

1. 오른쪽 위 모서리에서 **Admin**을 선택합니다.
1. 왼쪽 사이드바에서 **Geo** > **사이트**를 선택합니다.
1. 사용자 정의할 사이트에서 **편집**을 선택합니다.
1. 내부 URL을 편집합니다.
1. **변경 사항 저장**을 선택합니다.

활성화되면 **운영자** 영역은 Geo를 위해 프라이머리 사이트의 UI에서 직접 각 사이트의 복제 세부 정보를 표시하며, Geo 세컨더리 프록시가 활성화된 경우를 통해서도 표시됩니다.

> [!warning]
> Geo 사이트를 구성할 때 HTTPS 연결을 사용하는 것을 권장합니다. HTTPS를 사용할 때 **프라이머리**와 **세컨더리** 사이트 간의 통신이 끊어지는 것을 방지하려면 로드 밸런서에서 TLS가 종료되는 로드 밸런서를 가리키도록 내부 URL을 사용자 정의합니다.
