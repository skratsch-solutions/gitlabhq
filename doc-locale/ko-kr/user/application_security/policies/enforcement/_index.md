각 프로젝트 또는 그룹에 대한 새로운 보안 정책을 생성할 수 있지만, 여러 top-level groups에 걸쳐 동일한 정책 설정을 복제하는 것은 시간이 많이 걸릴 수 있으며 규정 준수 문제를 야기할 수 있습니다. 정책을 생성하기 전에 정책이 다음 중 어떤 것인지 알아야 합니다:

- 특정 프로젝트 또는 그룹에 적용됩니다.
- 여러 프로젝트에 적용합니다.
- 전체 인스턴스 또는 top-level group에 걸쳐 적용됩니다

정책을 여러 방식으로 적용할 수 있습니다:

- 단일 프로젝트 또는 그룹의 모든 프로젝트에 정책을 적용하려면 해당 프로젝트 또는 그룹에서 정책을 생성합니다.
- 여러 프로젝트에 걸쳐 정책을 적용하려면 [보안 정책 프로젝트](security_policy_projects.md)를 사용합니다. 보안 정책 프로젝트는 정책을 추가하고 그 외 다른 것은 없는 특수한 유형의 프로젝트입니다. 보안 정책 프로젝트의 정책을 다른 그룹 및 프로젝트에 적용하려면 그룹 또는 프로젝트에서 보안 정책 프로젝트에 연결합니다.
- GitLab Self-Managed 인스턴스 전체에 정책과 규정 준수 프레임워크를 함께 적용하려면 인스턴스 관리자가 [규정 준수 및 보안 정책 관리 그룹](compliance_and_security_policy_groups.md)을 사용할 수 있습니다.

## 정책 설계 지침 {#policy-design-guidelines}

정책을 설계할 때:

- 관리 오버헤드를 최소화하면서 범위를 최대화합니다
- 담당 업무 분리 보장

### 적용 {#enforcement}

요구 사항을 충족하기 위해 정책을 적용하려면 다음 요소를 고려하세요:

- **Inheritance**: 기본적으로 정책은 연결된 조직 단위와 모든 하위 그룹 및 해당 프로젝트에 적용됩니다.
- **Scope**: 정책 적용을 사용자 지정하려면 정책의 scope를 정의하여 요구 사항과 일치시킬 수 있습니다.

#### 상속 {#inheritance}

정책 범위를 최대화하려면 보안 정책 프로젝트를 목표를 달성하는 최상위 조직 단위(그룹, 하위 그룹 또는 프로젝트)에 연결합니다. 정책은 연결된 조직 단위와 모든 하위 그룹 및 해당 프로젝트에 적용됩니다. 최상위 수준에서의 적용은 필요한 보안 정책의 수를 최소화하여 관리 오버헤드를 줄입니다.

정책 상속을 사용하여 단계적으로 정책을 배포할 수 있습니다. 예를 들어 새로운 정책을 배포할 때 단일 프로젝트에 적용한 다음 테스트를 수행할 수 있습니다. 테스트를 통과하면 프로젝트에서 제거하고 그룹에 적용한 후 정책이 모든 해당 프로젝트에 적용될 때까지 계층 구조를 따라 올라갑니다.

기존 그룹 또는 하위 그룹에 적용된 정책은 다음이 충족되는 경우 그 아래에 생성된 새로운 하위 그룹 및 프로젝트에 자동으로 적용됩니다:

- 새로운 하위 그룹 및 프로젝트가 정책의 scope 정의에 포함됩니다(예: scope에 이 그룹의 모든 프로젝트가 포함됨).
- 기존 그룹 또는 하위 그룹이 이미 보안 정책 프로젝트에 연결되어 있습니다.

> [!note]
> GitLab.com 사용자는 최상위 그룹에 대해 또는 하위 그룹 전체에 걸쳐 정책을 적용할 수 있지만 GitLab.com top-level groups에 걸쳐 정책을 적용할 수는 없습니다. GitLab Self-Managed 관리자는 자신의 인스턴스의 여러 top-level groups에 걸쳐 정책을 적용할 수 있습니다.

다음 예제는 두 그룹 및 해당 구조를 보여줍니다:

- Alpha 그룹은 각각 여러 프로젝트를 포함하는 두 개의 하위 그룹을 포함합니다.
- Security and compliance 그룹은 두 개의 정책을 포함합니다.

**Alpha** 그룹(코드 프로젝트 포함)

- **Finance** (하위 그룹)
  - 프로젝트 A
  - 계정 수령(하위 그룹)
    - 프로젝트 B
    - 프로젝트 C
- **Engineering** (하위 그룹)
  - 프로젝트 K
  - 프로젝트 L
  - 프로젝트 M

**보안 및 규정 준수** 그룹(보안 정책 프로젝트 포함)

- 보안 정책 관리
- 보안 정책 관리 - 보안 정책 프로젝트
  - SAST 정책
  - Secret detection 정책

정책이 적용되지 않는 경우를 가정하고 다음 예제를 고려합니다:

- "SAST" 정책이 Alpha 그룹에서 적용되면 정책은 Alpha의 두 하위 그룹(Finance 및 Engineering)과 모든 프로젝트 및 하위 그룹에 적용됩니다. Secret detection 정책이 "계정 수령" 하위 그룹에서도 적용되면 두 정책이 프로젝트 B 및 C에 적용됩니다. 하지만 "SAST" 정책만 프로젝트 A에 적용됩니다.
- "SAST" 정책이 "계정 수령" 하위 그룹에서 적용되면 프로젝트 B 및 C에만 적용됩니다. 프로젝트 A에는 적용되는 정책이 없습니다.
- Secret detection 정책이 프로젝트 K에서 적용되면 프로젝트 K에만 적용됩니다. 다른 하위 그룹 또는 프로젝트에는 적용되는 정책이 없습니다.

#### Scope {#scope}

{{< history >}}

- GitLab 16.7에 [도입](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/135398)됨, [플래그](../../../../administration/feature_flags/_index.md) 포함, 이름: `security_policies_policy_scope`. 기본적으로 활성화됩니다.
- GitLab 16.11에서 [일반 공급](https://gitlab.com/gitlab-org/gitlab/-/issues/443594)됩니다. 기능 플래그 `security_policies_policy_scope`이 제거되었습니다.
- 그룹별 범위 설정 [도입](https://gitlab.com/gitlab-org/gitlab/-/issues/468384) GitLab 17.4.

{{< /history >}}

정책의 scope를 다음과 같이 세분화할 수 있습니다:

- 규정 준수 프레임워크: 선택된 규정 준수 프레임워크를 포함하는 프로젝트에 정책을 적용합니다.
- 그룹:
  - 그룹의 모든 프로젝트(그룹의 모든 하위 그룹 및 해당 프로젝트 포함). 특정 프로젝트를 선택적으로 제외합니다.
  - 여러 그룹의 모든 프로젝트(그룹의 모든 하위 그룹 및 해당 프로젝트 포함). 동일한 보안 정책 프로젝트에 연결된 모든 그룹을 정책에 나열할 수 있습니다. 특정 프로젝트를 선택적으로 제외합니다.
- 프로젝트: 특정 프로젝트를 포함하거나 제외합니다. 동일한 보안 정책 프로젝트에 연결된 프로젝트만 정책에 나열할 수 있습니다.

같은 정책에서 이러한 세분화를 함께 적용할 수 있습니다. 그러나 제외가 포함보다 우선합니다.

## 담당 업무 분리 {#separation-of-duties}

담당 업무 분리는 정책을 성공적으로 구현하는 데 필수적입니다. 개발 워크플로우를 지원하면서 필요한 규정 준수 및 보안 요구 사항을 충족하는 정책을 설계합니다.

담당 업무 분리를 구현할 때:

- 정책을 중앙에서 정의하고 개발 팀과 협력하여 정책이 해당 워크플로우를 지원하도록 합니다.
- 정책 수정 권한을 승인된 역할로만 제한합니다.

그룹, 하위 그룹 또는 프로젝트에 보안 정책 프로젝트를 적용하려면 다음 중 하나를 보유해야 합니다:

- 해당 그룹, 하위 그룹 또는 프로젝트의 Owner 역할.
- 해당 그룹, 하위 그룹 또는 프로젝트의 [custom role](../../../custom_roles/_index.md) (권한 포함): `manage_security_policy_link`.

Owner 역할 및 `manage_security_policy_link` 권한이 있는 custom roles는 그룹, 하위 그룹 및 프로젝트 전체에서 표준 계층 규칙을 따릅니다:

| 조직 단위 | 그룹 owner 또는 그룹 `manage_security_policy_link` 권한 | 하위 그룹 owner 또는 하위 그룹 `manage_security_policy_link` 권한 | 프로젝트 owner 또는 프로젝트 `manage_security_policy_link` 권한 |
|-------------------|---------------------------------------------------------------|---------------------------------------------------------------------|-------------------------------------------------------------------|
| 그룹             | {{< yes >}} | {{< no >}}  | {{< no >}} |
| 하위 그룹          | {{< yes >}} | {{< yes >}} | {{< no >}} |
| 프로젝트           | {{< yes >}} | {{< yes >}} | {{< yes >}} |
