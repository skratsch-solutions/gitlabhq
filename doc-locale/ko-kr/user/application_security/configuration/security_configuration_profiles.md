보안 구성 프로필은 프로젝트 전체에서 보안 스캐너가 실행되는 방식과 시기를 정의하는 중앙화된 설정입니다. 조직 전체에서 보안 스캐너를 효율적으로 관리하려면 보안 구성 프로필을 사용합니다. 프로필 기반 접근 방식은 최소한의 수동 설정으로 모범 사례를 적용합니다.

<i class="fa-youtube-play" aria-hidden="true"></i> 개요를 보려면 [보안 구성 프로필 소개](https://www.youtube.com/watch?v=XYMKhhtRvwA)를 참조하세요.

그룹에 프로필을 적용하면 해당 그룹 내의 각 개별 프로젝트에 적용됩니다. 프로필은 그룹 자체에 첨부되지 않으며 프로필이나 하위 그룹 간에 상속이 없습니다.

[기본 프로필](#default-profiles)을 사용하여 몇 분 안에 사전 구성된 보안 스캔을 활성화하고 최소한의 구성으로 설정합니다.

## 보안 스캐너 구성 {#configure-security-scanners}

프로필을 평가하고 관리하려면 그룹의 [보안 인벤토리](../security_inventory/_index.md#view-the-security-inventory)를 중앙 대시보드로 사용합니다.

### 테스트 범위 검토 {#review-test-coverage}

SAST, DAST 및 시크릿 검색과 같은 그룹의 스캐너에 대한 높은 수준의 상태(**활성화**, **비활성화** 또는 **실패함**)를 보려면:

1. 상단 표시줄에서 **검색 또는 이동**을 선택하고 그룹을 찾습니다.
1. 왼쪽 사이드바에서 **보안** > **보안 인벤토리**를 선택합니다.
1. 보안 인벤토리에서 **테스트 범위** 열을 검토합니다.

### 개별 프로젝트 범위 변경 {#change-individual-project-coverage}

특정 프로젝트를 구성하려면:

1. 상단 표시줄에서 **검색 또는 이동**을 선택하고 그룹을 찾습니다.
1. 왼쪽 사이드바에서 **보안** > **보안 인벤토리**를 선택합니다.
1. 프로젝트 옆에서 수직 줄임표({{< icon name="ellipsis_v" >}})를 선택하고 **Manage tool coverage**를 선택합니다.
1. 개별 스캐너를 켜거나 끕니다.

### 여러 프로젝트에 프로필 적용 {#apply-a-profile-to-multiple-projects}

시간을 절약하려면 여러 프로젝트에 보안 설정을 한 번에 적용할 수 있습니다:

1. 상단 표시줄에서 **검색 또는 이동**을 선택하고 그룹을 찾습니다.
1. 왼쪽 사이드바에서 **보안** > **보안 인벤토리**를 선택합니다.
1. 여러 프로젝트 또는 전체 하위 그룹을 선택하여 설정을 적용합니다.
1. **Bulk Action** 드롭다운을 선택하고 **보안 스캐너 관리**를 선택합니다.
1. **전체에 기본 프로필 적용**을 선택하여 선택한 항목 전체에서 보안 태세를 표준화합니다.

## 기본 프로필 {#default-profiles}

GitLab은 최소한의 구성으로 보안 스캔을 활성화할 수 있도록 사전 구성된 스캐너 설정인 기본 프로필을 제공합니다.

### 시크릿 검색 프로필 {#secret-detection-profile}

시크릿 검색 프로필을 적용하면 전체 개발 워크플로우에서 시크릿에 대한 권장 기본 보호를 활성화합니다. 프로필은 다음 스캔 트리거를 활성화합니다:

- **푸시 보호**:  모든 Git 푸시 이벤트를 스캔하고 시크릿이 감지된 푸시를 차단하여 시크릿이 코드베이스에 절대 들어가지 않도록 방지합니다.
- **머지 리퀘스트 파이프라인**:  열린 머지 리퀘스트가 있는 브랜치에 새 커밋이 푸시될 때마다 자동으로 스캔을 실행합니다. 결과는 머지 리퀘스트에서 도입된 새 취약점으로 범위가 지정됩니다. 모든 브랜치를 대상으로 합니다.
- **브랜치 파이프라인 (기본값 만)**:  변경 사항이 기본 브랜치에 병합되거나 푸시될 때 자동으로 실행되어 기본 브랜치의 시크릿 검색 태세에 대한 완전한 그림을 제공합니다. 모든 브랜치를 대상으로 합니다.

### SAST 프로필 {#sast-profile}

SAST 프로필을 적용하면 권장 구성을 사용하여 프로젝트 전체에서 정적 애플리케이션 보안 테스트를 활성화합니다. 프로필은 다음 스캔 트리거를 활성화합니다:

- **머지 리퀘스트 파이프라인**:  열린 머지 리퀘스트가 있는 브랜치에 새 커밋이 푸시될 때마다 자동으로 SAST 스캔을 실행합니다. 결과에는 머지 리퀘스트에서 도입된 새 취약점만 포함됩니다. 모든 브랜치를 대상으로 합니다.
- **브랜치 파이프라인 (기본값 만)**:  변경 사항이 기본 브랜치에 병합되거나 푸시될 때 자동으로 실행되어 기본 브랜치의 SAST 태세에 대한 완전한 그림을 제공합니다. 기본 브랜치를 대상으로 합니다.

### 프로필에 대한 세부 정보 보기 {#view-details-about-a-profile}

시크릿 검색 프로필에 대한 기술 세부 정보를 보려면:

1. 상단 표시줄에서 **검색 또는 이동**을 선택하고 그룹을 찾습니다.
1. 왼쪽 사이드바에서 **보안** > **보안 인벤토리**를 선택합니다.
1. **시크릿 검색** 프로필을 선택합니다.
1. 다음 정보를 검토합니다:
   - **분석기 유형**:  프로필의 유형(예: **시크릿 검색**, **SAST**)입니다.
   - **스캔 트리거**:  프로필이 지원하는 트리거(예: **푸시 보호**, **머지 리퀘스트 파이프라인**, **브랜치 파이프라인**)입니다.
   - **상태**:  프로필이 현재 범위에 대해 **활성**인지 **비활성화됨**인지를 범위 상태 표시기를 사용하여 표시합니다.

## 범위 상태 표시기 {#coverage-status-indicators}

시스템은 인벤토리의 시각적 큐를 사용하여 프로젝트 보호 여부를 표시합니다:

- **Solid green bar**:  스캐너는 완전히 활성화되고 활성입니다.
- **Gray/empty bar**:  스캐너가 아직 구성되거나 활성화되지 않았습니다.
- **Partial bar**:  일부 보호가 활성화되어 있습니다(예: 프로필에서 사용 가능한 일부 트리거는 활성화되지만 다른 트리거는 그렇지 않음).
- **Tooltips**:  범위 막대에 마우스를 가져가면 파이프라인 기반 스캔의 **last scan** 날짜와 특정 파이프라인 상태를 볼 수 있습니다.

파이프라인 기반 스캔과 달리 푸시 보호는 푸시 프로세스 중에 실시간으로 실행되기 때문에 마지막 스캔 날짜가 없습니다.

## 문제 해결 {#troubleshooting}

보안 구성 프로필로 작업할 때 다음 이슈가 발생할 수 있습니다.

### 푸시 보호에 마지막 스캔 날짜가 표시되지 않음 {#no-last-scan-date-appears-for-push-protection}

푸시 보호는 이벤트 기반이며 일정 기반이 아닙니다. `git push` 프로세스 중에 실시간으로 시크릿을 가로챕니다. `push` 명령 시점에 활성화되기 때문에 파이프라인 기반 스캐너에서 예상할 수 있는 것과 같은 마지막 스캔 날짜가 없습니다.

### 스캐너 상태는 대시보드에서 활성이지만 인벤토리 도구 설명에서 활성화되지 않음 {#scanner-status-is-active-in-the-dashboard-but-not-enabled-in-inventory-tooltip}

프로젝트가 레거시 설정을 사용하면서 동시에 새 프로필이 할당될 때 이 문제가 발생할 수 있습니다.

이 이슈를 해결하려면:

1. **Security Configuration** 페이지를 확인하여 가장 정확한 현재 프로필 상태를 확인합니다.
1. 필요한 경우 `.gitlab-ci.yml` 파일에서 레거시 스캐너 구성을 제거하여 프로필 기반 구성만 사용합니다.

> [!note]
> 인벤토리 도구 설명이 레거시 설정과 프로필 기반 설정 모두의 결합된 상태를 반영하도록 개선되고 있습니다.

### 레거시 대 프로필 기반 구성 이해 {#understanding-legacy-versus-profile-based-configuration}

레거시 스캐너 구성에서 프로필 기반 구성으로 마이그레이션하는 경우 다음 차이점을 참고하세요:

- 레거시 구성:  스캐너를 활성화하려면 YAML 파일 또는 개별 프로젝트 설정을 수동으로 편집해야 합니다.
- 프로필 기반 구성:  코드를 수정하지 않고도 한 번에 여러 프로젝트에 기본 프로필을 적용할 수 있는 중앙화된 시스템을 사용합니다.

프로젝트 전체에서 더 쉬운 관리와 더 큰 일관성을 위해 프로필 기반 구성이 권장됩니다.
