パイプライン実行ポリシーは、プロジェクトのパイプラインでカスタムCI/CDジョブを強制します。スケジュールされたパイプライン実行ポリシーを使用すると、この適用範囲を拡張して、定期的なケイデンス（毎日、毎週、または毎月）でCI/CDジョブを実行できます。これにより、コンプライアンススクリプト、セキュリティスキャン、その他のカスタムCI/CDジョブは、新しいコミットがない場合でも確実に実行されます。

## パイプライン実行ポリシーのスケジュール {#scheduling-your-pipeline-execution-policies}

既存のパイプラインにジョブを挿入または上書きする通常のパイプライン実行ポリシーとは異なり、スケジュールされたポリシーは、定義したスケジュールで独立して実行される新しいパイプラインを作成します。スケジュールされたパイプラインは、プロジェクトの`.gitlab-ci.yml`とは別のものであり、プロジェクトのCI/CDジョブは実行しません。

一般的なユースケースを次に示します:

- コンプライアンス要件を満たすために、定期的なケイデンスでセキュリティスキャンを強制します。
- プロジェクトの設定を定期的にチェックします。
- 非アクティブなリポジトリで依存関係のスキャンを実行し、新しく発見された脆弱性を検出します。
- コンプライアンスレポートスクリプトをスケジュールで実行します。

## スケジュールされたパイプライン実行ポリシーをテストする {#test-a-scheduled-pipeline-execution-policy}

すべてのプロジェクトでスケジュールされたパイプライン実行ポリシーを有効にする前に、テストを実行して、パイプラインが機能することを確認し、ポリシーがインフラストラクチャにどのような影響を与えるかを理解できます。テスト実行は実際のパイプラインを実行し、正確なタイミングとリソースの見積もりを提供します。

> [!note]
> テスト実行では実際のパイプラインが作成され、ターゲットプロジェクトに対してコンピューティング時間が消費されます。

テストを実行するには:

1. 左側のサイドバーで、**セキュリティ** > **ポリシー**を選択します。
1. テストするスケジュールされたパイプライン実行ポリシーを選択します。
1. **テスト実行**タブを選択します。
1. グループレベルでポリシーを表示している場合は、ドロップダウンリストからターゲットプロジェクトを選択します。
1. **テスト実行を開始する**を選択します。

テスト実行は、ポリシーのCI/CD設定を使用して、選択したプロジェクト上にパイプラインを作成します。**テスト実行**タブでテスト実行のステータスを監視できます。

テスト実行が完了すると、**テスト実行**タブに、実行時間やエラーメッセージなどの結果が表示されます。

## スケジュールパイプライン実行ポリシーを設定する {#configure-schedule-pipeline-execution-policies}

スケジュールされたパイプライン実行ポリシーを設定するには、セキュリティポリシープロジェクトの`.gitlab/security-policies/policy.yml`ファイルの`pipeline_execution_schedule_policy`セクションに設定フィールドを追加します:

```yaml
pipeline_execution_schedule_policy:
- name: Scheduled Pipeline Execution Policy
  description: ''
  enabled: true
  content:
    include:
    - project: your-group/your-project
      file: security-scan.yml
  schedules:
  - type: daily
    start_time: '10:00'
    time_window:
      value: 600
      distribution: random
```

### スケジュール設定スキーマ {#schedule-configuration-schema}

`schedules`セクションでは、セキュリティポリシージョブが自動的に実行されるタイミングを設定できます。特定の実行時間と配布期間を持つ、日次、週次、または月次のスケジュールを作成できます。

### スケジュール設定オプション {#schedules-configuration-options}

`schedules`セクションは以下のオプションをサポートします:

| パラメータ | 説明 |
|-----------|-------------|
| `type` | スケジュールタイプ: `daily`、`weekly`、または`monthly` |
| `start_time` | 24時間形式でのスケジュールの開始時刻（HH:MM） |
| `time_window` | パイプライン実行を分散する時間枠 |
| `time_window.value` | 期間（秒単位）（最小: 600、最大: 2629746） |
| `time_window.distribution` | 配布方法（現在、`random`のみがサポートされています） |
| `timezone` | IANAタイムゾーン識別子（指定しない場合はUTCがデフォルト） |
| `branches` | ブランチの名前を含むオプションの配列。これに対してパイプラインがスケジュールされます。`branches`が指定されている場合、パイプラインは指定されたブランチ上でのみ実行され、それらのブランチがプロジェクトに存在する場合に限られます。指定されていない場合、パイプラインはデフォルトブランチ上でのみ実行されます。スケジュールごとに最大5つの一意のブランチ名を指定できます。 |
| `days` | 週次スケジュールでのみ使用: スケジュールが実行される曜日（例: `["Monday", "Friday"]`） |
| `days_of_month` | 月次スケジュールでのみ使用: スケジュールが実行される日付の配列（例: `[1, 15]`、1から31までの値を含めることができます） |
| `snooze` | スケジュールを一時的に一時停止するオプションの設定 |
| `snooze.until` | スヌーズ後にスケジュールが再開されるISO8601形式の日時（フォーマット: `2025-06-13T20:20:00+00:00`） |
| `snooze.reason` | スケジュールがスヌーズされる理由を説明するオプションのドキュメント |

### スケジュールの例 {#schedule-examples}

日次、週次、または月次スケジュールを使用します。

#### 日次スケジュールの例 {#daily-schedule-example}

```yaml
schedules:
  - type: daily
    start_time: "01:00"
    time_window:
      value: 3600  # 1 hour window
      distribution: random
    timezone: "America/New_York"
    branches:
      - main
      - develop
      - staging
```

#### 週次スケジュールの例 {#weekly-schedule-example}

```yaml
schedules:
  - type: weekly
    days:
      - Monday
      - Wednesday
      - Friday
    start_time: "04:30"
    time_window:
      value: 7200  # 2 hour window
      distribution: random
    timezone: "Europe/Berlin"
```

#### 月次スケジュールの例 {#monthly-schedule-example}

```yaml
schedules:
  - type: monthly
    days_of_month:
      - 1
      - 15
    start_time: "02:15"
    time_window:
      value: 14400  # 4 hour window
      distribution: random
    timezone: "Asia/Tokyo"
```

### 時間枠の分散 {#time-window-distribution}

複数のプロジェクトにポリシーを適用する際にCI/CDインフラストラクチャが過負荷になるのを防ぐため、スケジュールされたパイプライン実行ポリシーは、いくつかの共通ルールに基づいて、パイプラインの作成を時間枠全体に分散します:

- すべてのパイプラインは`random`でスケジュールされます。パイプラインは、指定された時間枠内でランダムに分散されます。
- 最小時間枠は10分（600秒）、最大は約1か月（2,629,746秒）です。
- 月次スケジュールの場合、特定の日（2月31日など）に存在しない日付を指定すると、その実行はスキップされます。
- セキュリティポリシープロジェクトには、最大5つのスケジュールされたパイプライン実行ポリシーを含めることができます。
- スケジュールされたポリシーは、一度に1つのスケジュール設定のみを持つことができます。
- スケジュールされたポリシーは、最大5つのブランチをターゲットにできます。`branches`を省略した場合、ポリシーはプロジェクトのデフォルトブランチ上でのみ実行されます。
- 複数のプロジェクトにポリシーを適用する際は、利用可能なRunnerの容量に基づいて、プロジェクト数に対応できる十分な大きさの時間枠があることを確認してください。たとえば、1時間の時間枠で1000のプロジェクトに適用されるポリシーは、その1時間を通じてパイプラインの作成を均等に分散します（1分あたり約16のパイプライン）。Runnerがこのパイプライン作成レートを処理できることを確認するか、キューイングや遅延を避けるためにより大きな時間枠を選択してください。
- 月次スケジュールの場合、時間枠中のランダムな分散により、連続する実行間の間隔が異なる場合があります。たとえば、月次スケジュールは前回の実行から20日後に実行され、その後30日後に実行される場合があります。この分散は、インフラストラクチャ全体の負荷分散に役立つため、期待される動作です。

## スケジュールされたパイプライン実行ポリシーをスヌーズする {#snooze-scheduled-pipeline-execution-policies}

スヌーズ機能を使用して、スケジュールされたパイプライン実行ポリシーを一時的に一時停止できます。スヌーズ機能は、メンテナンス期間、休日、またはスケジュールされたパイプラインが特定の期間実行されないようにする必要がある場合に使用します。

### スヌーズの仕組み {#how-snoozing-works}

スケジュールされたパイプライン実行ポリシーをスヌーズすると:

- スヌーズ期間中は、新しいスケジュールされたパイプラインは作成されません。
- スヌーズ前に作成されたパイプラインは引き続き実行されます。
- ポリシーは有効なままですが、スヌーズ状態になります。
- スヌーズ期間の有効期限が切れると、スケジュールされたパイプライン実行は自動的に再開されます。

### スヌーズを設定する {#configuring-snooze}

スケジュールされたパイプライン実行ポリシーをスヌーズするには、スケジュール設定に`snooze`セクションを追加します:

```yaml
pipeline_execution_schedule_policy:
- name: Weekly Security Scan
  description: 'Run security scans every week'
  enabled: true
  content:
    include:
    - project: your-group/your-project
      file: security-scan.yml
  schedules:
  - type: weekly
    start_time: '02:00'
    time_window:
      value: 3600
      distribution: random
    timezone: UTC
    days:
      - Monday
    snooze:
      until: "2025-06-26T16:27:00+00:00"  # ISO8601 format
      reason: "Critical production deployment"
```

`snooze.until`パラメータは、ISO8601形式の`YYYY-MM-DDThh:mm:ss+00:00`を使用して、スヌーズ期間が終了する日時を指定します:

- `YYYY-MM-DD`: 年、月、日
- `T`: 日付と時刻の区切り
- `hh:mm:ss`: 時、分、秒（24時間形式）
- `+00:00`: UTCからのタイムゾーンオフセット（UTCの場合はZ）

例えば、`2025-06-26T16:27:00+00:00`は2025年6月26日午後4時27分（UTC）を表します。

### スヌーズの解除 {#removing-a-snooze}

スヌーズの有効期限が切れる前にスヌーズを解除するには、ポリシー設定から`snooze`セクションを削除するか、`until`値に過去の日付を設定します。

## 特定のブランチのパイプラインをスケジュールする {#schedule-pipelines-for-specific-branches}

デフォルトでは、スケジュールはデフォルトブランチ上でのみ実行されます。スケジュールされたパイプライン実行ポリシーはブランチフィルタリングをサポートしており、追加のブランチのパイプラインをスケジュールできます。`branches`プロパティを使用して、プロジェクト内の他の重要なブランチで定期的なスキャンやチェックを実行します。

スケジュールで`branches`プロパティを設定する場合:

- ブランチを指定しない場合、スケジュールされたパイプラインはデフォルトブランチ上でのみ実行されます。
- ブランチを指定した場合、ポリシーはプロジェクトに実際に存在する指定された各ブランチに対してパイプラインをスケジュールします。
- スケジュールごとに最大5つの一意のブランチ名を指定できます。
- 各ブランチ名は完全に指定する必要があります。ワイルドカードマッチングはサポートされていません。

### ブランチフィルタリングの例 {#branch-filtering-example}

```yaml
pipeline_execution_schedule_policy:
- name: Scan Multiple Branches
  description: 'Run security scans on main, staging and develop branches'
  enabled: true
  content:
    include:
    - project: your-group/your-project
      file: security-scan.yml
  schedules:
  - type: weekly
    days:
      - Monday
    start_time: '02:00'
    time_window:
      value: 3600
      distribution: random
    branches:
      - main
      - staging
      - develop
      - feature/new-authentication
```

この例では、指定されたすべてのブランチがプロジェクトに存在する場合、ポリシーは4つの個別のパイプライン（各ブランチに1つずつ）を作成します。

## 前提条件 {#prerequisites}

スケジュールされたパイプライン実行ポリシーを使用するには、プロジェクトが以下の要件を満たす必要があります:

- CI/CD設定ファイルは、以下のいずれかの場所に保存されています:
  - セキュリティポリシープロジェクト
  - パブリックプロジェクト
  - アクセスが有効になっているプライベートプロジェクト（[Enable access to CI/CD configuration files](#enable-access-to-cicd-configuration-files)を参照）
- CI/CD設定ファイルには、スケジュールされたパイプラインに適切なワークフロールールが含まれている必要があります。

## CI/CD設定ファイルへのアクセスを有効にする {#enable-access-to-cicd-configuration-files}

ポリシーがCI/CD設定ファイルを参照する場合、セキュリティポリシーボットはそれらにアクセスできる必要があります。パブリックプロジェクト内のファイルはデフォルトでアクセス可能です。セキュリティポリシープロジェクトまたはその他のプライベートプロジェクト内のファイルについては、以下のいずれかのオプションを使用してアクセスを有効にします。

### オプション1: セキュリティポリシープロジェクト内のファイルへのアクセスを許可する {#option-1-grant-access-to-files-in-the-security-policy-project}

CI/CD設定ファイルがセキュリティポリシープロジェクト自体に保存されている場合は、このオプションを使用します。この設定は、パイプライン実行ポリシーが注入されたパイプラインをトリガーするすべてのユーザーに適用されます。

1. セキュリティポリシープロジェクトの左側のサイドバーで、**設定** > **一般**を選択します。
1. **可視性、プロジェクトの機能、権限**を展開します。
1. **Grant security policy project access to CI/CD configuration**をオンにします。
1. **変更を保存**を選択します。

### オプション2: セキュリティポリシーボットがプライベートまたは内部プロジェクトにアクセスできるようにする {#option-2-allow-security-policy-bot-access-to-private-or-internal-projects}

ポリシーの`include:`値が、セキュリティポリシープロジェクト以外のプライベートまたは内部プロジェクトに保存されているCI/CD設定ファイルを参照する場合は、このオプションを使用します。この設定は、セキュリティポリシーボットユーザーにのみ適用され、任意のプロジェクトで有効にできます。

1. セキュリティポリシープロジェクトで`pipeline_execution_policy_bot_access`実験を有効にします。`.gitlab/security-policies/policy.yml`ファイルに、以下の行を追加します:

   ```yaml
   experiments:
     pipeline_execution_policy_bot_access:
       enabled: true
   ```

   > [!note]
   > あなたのプライベートまたは内部プロジェクト、あるいはその親グループのいずれかが、このセキュリティポリシープロジェクトにリンクされている必要があります。まだリンクされていない場合は、[セキュリティポリシープロジェクトをリンク](enforcement/security_policy_projects.md#link-to-a-security-policy-project)する必要があります。

1. CI/CDファイルを保存するプライベートまたは内部プロジェクトで、左側のサイドバーから**設定** > **一般**を選択します。
1. **可視性、プロジェクトの機能、権限**を展開します。
1. **セキュリティポリシーボットのアクセス**で、**Allow security policy bots to access CI/CD configuration files in this project**を選択します。
1. **許可するファイルパターン**で、ボットがアクセスできるファイルを指定するglobパターンを1つ以上追加します。複数のパターンはカンマで区切ります。
1. オプション。**許可されたグループ**で、そのグループ内のプロジェクトからのセキュリティポリシーボットのみがCI/CD設定ファイルにアクセスできるように、グループを選択します。

   指定しない場合、ルート祖先グループ内の任意のプロジェクトからのボットはファイルにアクセスできます。
1. **変更を保存**を選択します。

許可されるファイルのグロブパターンは、`include:file:`値で指定されたパスと一致する必要があります。例: 

- `include:file: ci/security-scan.yml`の場合は、`ci/**/*.yml`または`ci/security-scan.yml`を使用します。
- `include:file: policy-ci.yml`の場合は、`*.yml`または`policy-ci.yml`を使用します。
- 複数のディレクトリの場合は、`ci/**/*.yml, templates/**/*.yml`のように、カンマで区切られた複数のパターンを使用します。

## セキュリティポリシーボットユーザー {#security-policy-bot-user}

スケジュールされたパイプラインは、セキュリティポリシーが適用される各プロジェクトに対してGitLabが自動的に作成する専用のシステムアカウントであるセキュリティポリシーボットユーザーによって実行されます。ポリシーの実行が分離され、安全であることを保証するため、ボットユーザーには以下のセキュリティ制限があります:

- ボットユーザーは、その特定のプロジェクトのメンバーのみです。グループや他のプロジェクトに追加することはできません。
- ボットユーザーは外部ユーザーとして扱われ、デフォルトでは内部プロジェクトにアクセスできません。
- ボットユーザーは、セキュリティポリシープロジェクトおよびパブリックプロジェクト内のファイルにアクセスできます。
- ボットユーザーは、それらのプロジェクトが明示的に**セキュリティポリシーボットのアクセス**を有効にしており、ファイルパスがプロジェクトで指定されたパターンと一致する場合にのみ、プライベートまたは内部プロジェクトのファイルにアクセスできます。

ボットユーザーは他のプロジェクトのメンバーではないため、以下のいずれのアクションも完了できません:

- ボットアクセスを許可しない、または許可されたファイルパターンと一致しないプライベートまたは内部プロジェクトからのCI/CD設定ファイルへのアクセス。
- プライベートまたは内部プロジェクトをターゲットとするマルチプロジェクト子パイプラインを開始する。
- プライベートまたは内部プロジェクトからのアーティファクトまたはリソースへのアクセス。

> [!important]
プライベートまたは内部プロジェクトからファイルを含める場合、そのプロジェクトで**セキュリティポリシーボットのアクセス**を有効にし、一致するファイルパターンを設定してください。これらの設定がない場合、パイプライン実行はアクセスエラーで失敗します。

## スケジュール制限 {#scheduling-limits}

この機能はベータ版であり、今後のリリースで変更される可能性があります。スケジュールされたパイプライン実行ポリシーを作成する際は、以下の制限に注意してください:

- セキュリティポリシープロジェクトあたりのスケジュールされたパイプライン実行ポリシーの最大数は、1つのスケジュールを持つ1つのポリシーに制限されています。
- スケジュールの最大頻度は1日1回（日次）です。
- ブランチが指定されていない場合、スケジュールされたパイプライン実行ポリシーはデフォルトブランチ上でのみ実行されます。
- `branches`配列には、最大5つの一意のブランチ名を指定できます。
- パイプラインの十分な分散を確保するために、時間枠は少なくとも10分（600秒）である必要があります。
- Runnerが不足している場合、スケジュールされたパイプラインは遅延する可能性があります。

## トラブルシューティング {#troubleshooting}

スケジュールされたパイプラインが期待どおりに実行されない場合は、以下のトラブルシューティング手順に従ってください:

1. **ポリシーアクセスの確認**:次の点を確認します。
   - CI/CD設定ファイルは、セキュリティポリシープロジェクト、パブリックプロジェクト、またはボットアクセスが有効でファイルパターンが一致するプライベートまたは内部プロジェクトにあります。
   - セキュリティポリシープロジェクトで**パイプライン実行ポリシー**設定が有効になっています（**設定** > **一般** > **可視性、プロジェクトの機能、権限**）。
1. **CI設定の検証**:
   - 指定されたパスにCI/CD設定ファイルが存在することを確認します。
   - 手動でパイプラインを実行して、設定が検証するであることを確認します。
   - 設定に、スケジュールされたパイプラインに適切なワークフロールールが含まれていることを確認します。
1. **ポリシー設定の確認**:
   - ポリシーが有効になっていることを確認します（`enabled: true`）。
   - スケジュール設定が正しい形式と有効な値であることを確認します。
   - ブランチを指定した場合は、そのブランチがプロジェクトに存在することを確認します。
   - タイムゾーン設定が正しいことを確認します（指定されている場合）。
1. **ログとアクティビティの確認**:
   - セキュリティポリシープロジェクトのCI/CDパイプラインログでエラーがないか確認します。
1. **Runnerの可用性の確認**:
   - Runnerが利用可能で、適切に設定されていることを確認します。
   - Runnerがスケジュールされたジョブを処理する能力があることを確認します。
